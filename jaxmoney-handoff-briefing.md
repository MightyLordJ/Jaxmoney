# JaxMoney 專案交接資訊

## 專案是什麼
一個叫 **JaxMoney** 的網頁 App(PWA),給我自己用,主要功能:
- **換匯計算機**:5 個可自訂的幣別欄位 + 計算機鍵盤,可以互相換算,還有打折試算功能(長按 %$ 按鈕切換「折扣%」跟「打幾折」兩種模式)
- **歷史匯率**:查詢兩個幣別之間的歷史匯率走勢圖(1個月/半年/1年/3年),有收藏常用幣別對的功能
- 兩個頁面之間用**左右滑動**切換(不是底部分頁按鈕,那個已經拿掉了)
- 支援離線查看上次更新的匯率資料
- 資料來源:Frankfurter API(`https://api.frankfurter.dev/v2`),免費不用金鑰

## 技術架構
- 純 HTML/CSS/JS 單一檔案(`index.html`),沒有用任何框架
- 另外還有 `sw.js`(Service Worker,負責離線快取,**v3.6.0 起改成 stale-while-revalidate**——有快取就先秒開,背景再打網路更新快取,不再是每次開啟都先等網路;真正離線時才會退回快取,連 index.html 都拿不到的話才顯示離線畫面)跟 `apple-touch-icon.png`(App 圖示)
- 部署在 **GitHub Pages**,網址固定是 `https://mightylordj.github.io/miniproject/`
- 我是用「加入主畫面」的方式當作 App 使用(iOS PWA,standalone 模式),不是一般開分頁瀏覽

## 我的核心訴求 / 偏好
1. **深色 Apple 風格介面**,紫色(#5E5CE6)為主色調
2. **不要出現任何捲軸**,不管手機螢幕多小多大,內容都要剛好塞進螢幕
3. 標題列(JaxMoney 那排)要**固定在最上面**,不能因為下面內容變化就跟著移動
4. 兩個頁面用**滑動切換**,滑動時畫面要**跟著手指即時移動**(不是滑到底才突然跳),拖曳夠長才真的切換頁面,沒拖夠會彈回原本頁面
5. 兩個小圓點(頁面指示器)要**貼齊螢幕最下緣**
6. 每次開啟 App,選過的 5 個幣別要記得(存在 localStorage)
7. **小數點位數依幣別最小單位決定**,不是統一固定位數:
   - 沒有小數位(0 位):TWD、JPY、KRW、VND、ISK、HUF、IDR
   - 其餘幣別(USD、EUR、GBP...等)預設 2 位小數
   - 換匯計算機裡不管是「非 active 欄位顯示」「切換 active 欄位換算」「+ − × ÷ 計算結果」「打折試算結果」,都要套用同一套規則,不能各自為政(之前就出過 bug:非 active 欄位跟切換後的 active 欄位小數位數不一致,例如日圓從 4864 一切換變成 4864.4982)
   - 對照表寫在 `CURRENCY_DECIMALS` 這個物件裡,之後要調整哪個幣別該有幾位小數,改這裡就好

## ⚠️ 最核心、也是最頭痛的未解問題

**現象:** 這是一個 iOS Safari/WebKit 已知的 PWA 冷啟動 bug——App 長時間關閉(或背景待機一段時間)後重新打開,有一定機率(過去測試大概抓到快一半)畫面會判斷錯誤,導致整個內容被誤判成需要縮小 (RWD 縮放算錯),看起來畫面整個縮水、四周留白。**重新關掉再打開通常就會恢復正常**,但下次長時間關閉後又有機率再發生。

**根本原因(目前最可信的解釋):** iOS 在 App 從背景喚醒、或是 standalone 模式冷啟動的瞬間,`window.innerHeight` / `dvh` / `svh` 這些回報視窗高度的數值,有機率會回傳「假網址列還沒收起前」的暫時性錯誤數值,而我們的版面(不管用什麼縮放方式)如果剛好在那個空窗期做了量測跟計算,就會算錯。這個錯誤是**暫時性、會自己恢復**的,只是我們量測的時間點可能剛好卡在那個空窗期。

**已經試過、沒有解決或造成新問題的做法(避免重複踩雷):**
- 换成 `dvh`、`svh` 等新版 CSS 視窗單位——沒有徹底解決,`svh` 本身有時也一樣會被誤判
- 用 `window.screen.height`(硬體螢幕解析度)取代量測視窗高度——理論上該繞開問題,實測沒有明顯改善
- 完全拿掉縮放機制、改用可捲動的原始大小版面——使用者覺得小螢幕上太大、不能接受
- 把整個中間內容區包成一塊「固定設計尺寸畫布」用 `transform: scale()` 等比縮放——這個機制本身容易出現「量到 0 或極端值 → 縮放鎖死」的連鎖 bug,过程中製造了好幾次「每次打開都變超小」的災難性回歸,要非常小心處理極端值防呆
- 在 CSS 上疊加多層 `min-height` / `vh` / `dvh` / `calc(var())` ——容易互相打架,建議精簡成單一、清楚的來源

**目前採用的做法(v3.3.2,還在觀察成效階段):**
不強求「一次量準」,而是在 App 啟動、從背景恢復、視窗大小改變等時機,於接下來的 **2 秒內重複量測 8 次**(0/80/160/300/500/800/1200/1800 毫秒),只要 iOS 內部的過渡動畫一結束、數值恢復正常,就會自動套用修正,不需要使用者手動關掉重開。這個做法還在測試「長時間關閉後重開」的實際成功率,還沒有確認是否真正解決。

**如果這次還是沒解決,可以考慮的方向:**
- 進一步拉長重試的時間窗口或次數
- 研究是否能用 `ResizeObserver` 監控 `document.body` 或關鍵容器的實際尺寸變化,取代單純的定時重試
- 死心接受「完全無法根除,只能降低機率」,把重心放在使用者體感上(例如淡入動畫遮住修正瞬間的畫面跳動,而不是繼續硬解)
- 考慮包裝成真正的原生 App(用 Capacitor 之類的工具),因為 WKWebView 沒有這個「假網址列」的判斷邏輯,可能從根本上避開這整類問題(但這是最後手段,需要額外的開發環境跟蘋果開發者帳號費用)

**追加發現(v3.5.0):** 後來發現這個縮小 bug 其實有兩種不同成因混在一起討論:
1. 原本講的 iOS 假網址列瞬間錯誤值(冷啟動/背景喚醒觸發)
2. **另一個獨立成因:** `fitAppCanvas()` 是用 `canvas.scrollHeight`(實際內容高度)去算縮放比例,但匯率 API 資料是非同步載入的。如果 API 拖很久、超過原本 8 次定時重試的 1.8 秒窗口才回來,8 次重試會全部在內容還沒長好之前就量完,量到錯誤高度後,1.8 秒後也不會再重新校準,錯的縮放比例就卡住了。**這正好解釋了「載入越久越容易發生」這個現象**——跟原本猜的 iOS 瞬間錯誤值是兩回事。

已針對第 2 點加上 `ResizeObserver`(觀察 `#appCanvas` 跟 `.canvas-wrap` 的實際尺寸變化,一有變化就自動重跑 `fitAppCanvas()`),不用再猜 API 幾秒後才回來,理論上能徹底關掉這條路徑。原本的 8 次定時重試機制沒有拿掉,繼續處理第 1 點(iOS 瞬間錯誤值)。

**追加發現(v3.8.0):** 後來實測發現,連「關掉立刻重開」都會出現縮小,代表主因還是第 1 點(iOS 假過渡動畫瞬間錯誤值),而且比原本以為的更容易觸發,`ResizeObserver` 並沒有完全接住這個情況。因為這個成因目前還沒有可靠的正面解法,改用「體感掩蓋」的方式處理:加了一個啟動用的 splash 畫面(單純置中顯示 App 圖示,不套用 `transform:scale()`,所以不受縮放 bug 影響)。App 一開啟先強制顯示 splash 至少 700ms,這段時間內背景照常跑原本的定時重試 + `ResizeObserver`;700ms 後才做一次「正式」的 `fitAppCanvas()` 校正,再淡出 splash 露出真正的計算機主頁面。用意是把最容易踩到 iOS 那個「假過渡動畫」的量測時間點,往後延到一個相對安全的時機,而不是繼續正面硬解 iOS 本身的計時問題。如果之後 splash 消失後偶爾還是會看到一瞬間的縮放跳動,可以考慮拉長 `SPLASH_MIN_MS`(目前 700ms,在 `index.html` 底部初始化區塊裡)。

**桌面圖示偶爾變黑塊的問題:** 已確認 `apple-touch-icon.png` 本身沒問題(180×180、RGB、無透明色版、無異常色彩描述檔),排除是圖片檔案本身的問題。目前推測跟縮放 bug 是類似性質的成因——重開機後系統/網路還沒完全就緒,剛好卡到 iOS 重新產生桌面圖示的時間點,抓取失敗才顯示黑塊,屬於系統層級的行為,網頁程式碼這邊能做的有限,已加上明確的 `sizes="180x180"` 屬性降低誤判機率,但無法保證根治。
